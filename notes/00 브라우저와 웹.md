## 웹의 역사

- 초창기 인터넷
  - 주로 대학, 연구소, 주요 기업을 물리적 케이블로 연결한 전 세계 네트워크
  - 애플리케이션마다 서로 다른 프로토콜로 통신
  - 웹페이지는 컴퓨터에 저장된 특정 형식의 파일
  - 웹페이지의 URL은 특정 컴퓨터와 특정 파일 이름을 가리킴
  - 초기 서버는 디스크에서 파일을 읽어주는 것
  - 웹의 논리적 구조는 서버의 물리적 구조를 있는 그대로 반영
- 현대 인터넷
  - HTML은 서버에서 동적으로 조합되어 (SSR) 동적 컨텐츠가 채워지고 요청이 있을 때 브라우저로 전송
  - CDN (Content Delivery Network, 콘텐츠 전송 네트워크)가 URL을 전 세계의 서버 중 어디로든 라우팅
  - 클라우드 컴퓨팅 서비스에서 웹페이지 제공
- 웹의 핵심 요소
  - 웹: 하이퍼링크로 연결된 정보 네트워크
  - 사용자는 사용자 에이전트인 브라우저로 웹을 탐색
  - 정보는 HTTP 네트워크 프로토콜로 요청되며 HTML 문서 형식으로 구조화
  - 문서는 콘텐츠가 아닌 URL로 식별되며 동적으로 생성
  - 웹페이지는 이미지, 동영상, CSS, 자바스크립트 등 다양한 형식의 서브 리소스에 링크 가능
  - 모든 요소가 개방적이고 표준화되어 있으며 자유롭게 사용하거나 재사용 가능
    <br/>

## 실제 브라우저 코드베이스

- 브라우저가 다른 거대한 코드베이스와 다른 점: 긴급성
- 브라우저에서 문제가 발생하면 바로 처리해야 함 ➡️ 브라우저를 처음부터 다시 만드는 것보다 고치고 개선해야 함
- 많은 사람들이 브라우저 코드베이스에 기여하고 있으며, 여전히 기여할 여지가 많음
  <br/>

## 브라우저 코드 개념

- HTML, CSS는 선언적 API ➡️ 그 API를 구현하는 것은 브라우저의 역할
- 브라우저를 개발하려면 제어의 역전 (inversion of control), 제약 프로그래밍 (constraint programming), 선언형 프로그래밍 (declarative programming)을 구현해야 함 ➡️ 개발자는 어느 정도 통제권과 주도권을 잃음
  - 개발자가 렌더링에 필요한 매개변수와 콘텐츠를 브라우저에 전달하면, 브라우저는 제어를 역전해 대부분의 렌더링을 처리
  - 이때 대부분의 매개변수는 화면의 직접적인 위치보다는 상대적인 크기와 위치 같은 제약 조건의 형태로 주어짐, 브라우저는 이 제약 조건을 풀어서 값을 계산
  - 선언적 스타일의 웹페이지 이벤트는 개발자 관점에서는 즉시 적용되는 것처럼 보이지만, 브라우저 내부에서는 실제로 연산 수행을 최대한 지연시켜 다른 API 호출이나 사용자에게 페이지가 보여야 할 때가 되서야 적용될 수 있음

<br/>

## 브라우저의 역할

- 샌드박스: 웹 브라우징을 안전하게 보호
- 알고리즘: 선언적 문서 모델을 구현
- UI: 링크 탐색을 지원
- 브라우저가 극도로 효율적일 때만 웹페이지가 빠르게 로드되고 원활하게 반응

### 메멕스 (Memex)

- "As We May Think" - Vannevar Bush
- 메멕스 (Memex)
  - 개인에게 전 세계의 정보를 열람하고 탐색할 수 있도록 도와주는 기계
  - 기능: 기록된 정보의 조회 + 기록 간의 연결
- 메멕스와 웹의 핵심
  - 인간이 효율적으로 배우고 탐색하는 방법을 제공하고, 정보를 표현하고 나타내는 것
  - 인류의 집단지능과 지혜에 빠르게 접근 + 정보와 지식을 정리(연결)하여 유용하게 만드는 것
- 연관: 큐레이션한 링크를 통해 하나의 정보에서 다른 정보로 이어지는 생각의 흐름

### 웹의 등장

- 하이퍼링크로 연결되는 하이퍼텍스트 문서의 개념 등장
  - 하이퍼텍스트: 하이퍼링크로 다른 텍스트와 연결된 텍스트 (웹페이지)
  - 하이퍼링크: 웹페이지 사이의 연결
  - HTML: 웹페이지를 작성하는 포맷
  - HTTP: 웹페이지를 로드하는 프로토콜
- 웹의 핵심 개념과 브라우저 UI의 핵심 요소 소개
- 최초의 웹 브라우저와 웹 서버 탄생
  - 웹의 탈 중앙적인 성격
  - ➡️ 게이트키퍼가 없어 누구나 쉽게 간단한 웹페이지 개시 가능
  - ➡️ 페이지가 단순히 URL로 식별되었기 때문에 외부 소스에서 웹으로 트래픽 유입 가능
  - ➡️ 사이트 간 호환성과 하이퍼링크로 인한 네트워크 효과
  - ➡️ 표준 단체를 통해 유지되기 때문에 독점적 통제와 조작의 문제에서 자유로움

### 브라우저

- 최초로 널리 배포된 브라우저: ViolaWWW

- 마이크로소프트의 인터넷 익스플로러
- 첫 번째 브라우저 전쟁
  - 넷스케이프 내비게이터와 인터넷 익스플로러 간 치열한 경쟁
  - CSS, DOM, JS 등이 브라우저에 포함
- 두 번째 브라우저 전쟁
  - 다양한 브라우저들 간의 점유율 경쟁
  - AJAX 요청의 광범위한 사용, `<canvas>`같은 HTML5 기능, JS 라이브러리 및 프레임워크의 폭팔적인 증가

### 웹 표준

- 두 번째 브라우저 전쟁에서 모든 주류 브라우저가 크로미움, 게코, 웹킷이라는 세 가지 오픈 소스 기반의 웹 렌더링 엔진/자바스크립트 엔진에 기반해서 만들어짐
- 오픈 소스로 되어 있는 모든 주요 브라우저가 표준화 프로세스에 피드백을 제공하면서 웹의 탈중앙적 특징을 강화
